Dziennik API

Trzy endpointy do pracy z dziennikiem programowo: dodaj wpis, zapytaj agenta, odczytaj dzień. Per-user, autoryzowane długoterminowym tokenem.

Base URL

base 
https://kkegymepatwufnemtldr.supabase.co/functions/v1

Quick start

curl 
curl -X POST $URL/add-entry \
  -H "Authorization: Bearer dzn_xxx" \
  -H "Content-Type: application/json" \
  -d '{"content":"hello"}'
Wszystkie endpointy są edge functions wdrożone na Supabase. Daty domyślne (date nie podane) używają strefy Europe/Warsaw.

Autoryzacja

Każde żądanie musi zawierać nagłówek Authorization z tokenem API. Tokeny mają format dzn_<base64url> i są związane z jednym kontem użytkownika.

header 
Authorization: Bearer dzn_<twoj_token>

Serwer akceptuje też Supabase JWT (używane przez frontend) — token API jest dla integracji i automatyzacji.

Tokeny generuje się dziś ręcznie przez SQL w Supabase Studio (insert do public.api_tokens z sha256(token) w kolumnie token_hash). Endpoint do generacji z UI nie jest jeszcze gotowy.

Błędy & limity

Wszystkie błędy zwracają JSON w formacie { "error": "..." } z odpowiednim kodem HTTP.

StatusZnaczenie
400Niepoprawne body lub parametry (np. zły format daty, content poza 1–500 znakami).
401Brak nagłówka Authorization, zły token, lub token cofnięty.
405Zła metoda HTTP.
409Konflikt — dla Create oznacza osiągnięty limit 5 wpisów na dzień.
500Błąd po stronie serwera (np. brak sekretu, awaria bazy).
502Tylko Ask — model AI nieosiągalny.

Limit twardy: 5 wpisów na dzień kalendarzowy per user. Wymuszany w warstwie aplikacji.

Create — dodaj wpis

POST /add-entry

Tworzy nowy wpis dla autoryzowanego użytkownika. Pozycja (1–5) przydzielana automatycznie. Opcjonalnie zapisuje też mood 1–5 dla danego dnia (mood żyje na poziomie dnia, nie wpisu).

Body

FieldTypeOpis
contentrequired string Treść wpisu, 1–500 znaków po trim().
dateoptional string (YYYY-MM-DD) Data wpisu. Domyślnie dzisiaj w strefie Europe/Warsaw.
moodoptional integer 1–5 Nastrój dnia. Upsert do day_moods (nadpisuje wcześniejszą wartość dla tego dnia).

Request

curl 
curl -X POST $URL/add-entry \
  -H "Authorization: Bearer dzn_xxx" \
  -H "Content-Type: application/json" \
  -d '{
    "content": "dobry dzien, zbudowalem skilla",
    "mood": 5
  }'

Response

200 OKWpis utworzony
json
{
  "id": "d66aa9dd-8772-4b3f-9826-18a1762d154e",
  "date": "2026-06-23",
  "position": 2,
  "content": "dobry dzien, zbudowalem skilla",
  "mood": 5
}
409 ConflictLimit 5 wpisów na dzień
json
{
  "error": "Limit 5 wpisow na dzien osiagniety.",
  "date": "2026-06-23",
  "count": 5
}

Ask — zapytaj agenta

POST /notes-chat

Zadaje pytanie agentowi (Albert Einstein, model grok-3). Agent ma dostęp do tool-callu getNotesForRange i może czytać Twoje wpisy z dowolnego zakresu dat. Wpisy z dnia date (lub dzisiejszego) są wstrzykiwane do kontekstu automatycznie.

Body (single-shot)

FieldTypeOpis
questionrequired string Pojedyncze pytanie do agenta.
dateoptional string (YYYY-MM-DD) "Aktywny" dzień — wpisy z tego dnia trafiają do kontekstu systemowego. Domyślnie dziś.
Endpoint akceptuje też pełny format { messages, context } używany przez UI z historią konwersacji.

Request

curl 
curl -X POST $URL/notes-chat \
  -H "Authorization: Bearer dzn_xxx" \
  -H "Content-Type: application/json" \
  -d '{
    "question": "co robilem w zeszly piatek?",
    "date": "2026-06-19"
  }'

Response

200 OK
json
{
  "reply": "W piątek 2026-06-19 zapisałeś dwa wpisy...",
  "proposals": []
}

proposals to lista propozycji nowych wpisów wygenerowanych przez agenta tool-callem proposeNote. Agent nie zapisuje wpisów sam — jeśli chcesz je przyjąć, użyj Create.

Read — odczytaj dzień

GET /get-entry?date=YYYY-MM-DD

Zwraca wszystkie wpisy z danego dnia (posortowane po position rosnąco) oraz mood, jeśli ustawiony.

Query

ParamTypeOpis
dateoptional string (YYYY-MM-DD) Data do odczytania. Domyślnie dzisiaj w strefie Europe/Warsaw.

Request

curl 
curl "$URL/get-entry?date=2026-06-23" \
  -H "Authorization: Bearer dzn_xxx"

Response

200 OK
json
{
  "date": "2026-06-23",
  "notes": [
    {
      "id": "3c131afc-...",
      "position": 1,
      "content": "Nastrój 5/5 — duma, zbudowałem skilla.",
      "created_at": "2026-06-23T07:18:57.552Z",
      "updated_at": "2026-06-23T07:18:57.552Z"
    }
  ],
  "mood": 5
}

mood jest null, jeśli dla danego dnia nie ustawiono nastroju.

Tokeny API

Tokeny są przechowywane jako sha256(token) w hex w kolumnie token_hash. Surowy token widoczny jest tylko w momencie generacji.

Generacja tokenu (na razie ręcznie)

powershell
# 1. Wygeneruj losowy token i jego sha256
$bytes = New-Object byte[] 32
[System.Security.Cryptography.RandomNumberGenerator]::Create().GetBytes($bytes)
$b64 = [Convert]::ToBase64String($bytes).Replace('+','-').Replace('/','_').TrimEnd('=')
$token = "dzn_$b64"
$sha = [System.Security.Cryptography.SHA256]::Create()
$hash = ($sha.ComputeHash([Text.Encoding]::UTF8.GetBytes($token)) | % { $_.ToString('x2') }) -join ''
"TOKEN=$token"; "HASH=$hash"
sql
-- 2. Wstaw rekord (Supabase Studio > SQL Editor)
insert into public.api_tokens (user_id, token_hash, name)
values ('<twoje user_id>', '<HASH z kroku 1>', 'cli');

Cofnięcie tokenu: update api_tokens set revoked_at = now() where name = 'cli';
